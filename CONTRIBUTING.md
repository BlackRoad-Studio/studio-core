## ğŸ’» Development Workflow

### Branch Naming Convention

- `feature/` - New features
- `bugfix/` - Bug fixes
- `hotfix/` - Critical fixes for production
- `docs/` - Documentation updates
- `refactor/` - Code refactoring
- `test/` - Test additions or modifications

### Commit Message Convention

We follow conventional commits:

- `feat:` - New feature
- `fix:` - Bug fix
- `docs:` - Documentation changes
- `style:` - Code style changes (formatting, etc.)
- `refactor:` - Code refactoring
- `test:` - Test changes
- `chore:` - Maintenance tasks

Example: `feat: add new light system workflow`

## ğŸ“ Pull Request Process

1. **Choose the right template**: Select GreenLight, YellowLight, or RedLight based on your PR status
2. **Fill out the template**: Provide all requested information
3. **Self-review**: Review your own code before requesting review
4. **Pass CI checks**: Ensure all automated checks pass
5. **Request review**: Tag appropriate reviewers
6. **Address feedback**: Respond to all review comments
7. **Merge**: Once approved, your PR will be merged
